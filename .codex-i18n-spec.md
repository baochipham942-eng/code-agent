# Settings 内容区 i18n 迁移规范（批次施工单）

你在 worktree `code-agent-settingscontent`（分支 chore/settings-content-i18n）工作。
任务：把指定文件里的中文 UI 字面量迁移到 i18n，组件改读 `t`。

## 硬规则（违反任何一条 = 返工）

1. **纯文案搬迁，零行为改动**：不改逻辑/props/id/className/数据流，不重构，不新增抽象，不动与文案无关的任何行。
2. **zh 值逐字保留原文**：`src/renderer/i18n/zh.ts` 里的值必须与源码原字符串完全一致（含标点/空格）。
3. **en 同步补齐**：`en.ts` 是 `Translations`（由 zh 推导）类型，缺键会 typecheck 报错。英文翻译要自然、面向非程序员用户。
4. **键放对命名空间**：设置页内容键放 `settings.<tab命名空间>.*`（zh.ts 里已有 settings.general / settings.appearance 等分组，先看现有结构再加；该 tab 没有分组就新建 `settings.<tabId>: {...}`）。同一字符串已有现成键的直接复用，不要造重复键。
5. **组件内取 t**：组件已 import `useI18n` 的直接用；没有的加 `import { useI18n } from '<相对路径>/hooks/useI18n';` + `const { t } = useI18n();`。模块级（组件外）函数需要文案时优先把文案查找挪进组件作用域或传参（参考 slashPickerModel 的 labels 注入模式）；实在过于侵入就保留原样并在报告中列出。
6. **插值**：`` `共 ${n} 条` `` 这类用模板串拼 t 键（如 `` `${t.x.prefix}${n}${t.x.suffix}` ``）或拆前后缀键，禁止在 i18n 值里存变量名。
7. **禁止 git commit / push**——提交由监工做。
8. 中文**注释**不用迁移（棘轮扫描已排除注释）。
9. **孤儿命名空间**：如果组件原本在用某个顶层 i18n 命名空间（如 `t.xxx.*`），
   迁移到 `settings.<tab>` 后旧命名空间若无其他消费者（grep src + tests 确认），
   把旧命名空间从 zh.ts/en.ts 一并删除，避免重复文案；有其他消费者则保留并在报告说明。
10. **严禁用 `\uXXXX` 转义等任何手段绕过棘轮扫描**（棘轮已能识别转义，绕闸=返工）。
    组件里如果有对 zh 数据（如 shared 层中文契约）做字符串处理的逻辑，把该逻辑
    移到数据所在文件里导出函数（那里不在棘轮清单），渲染层只调函数；shared 层
    的中文数据本身不在本工程 scope，报告里列出即可。缩进跟随文件原有风格（空格），
    不要引入 TAB。

## 每个文件完成后的自验（必须全绿才算完）

```bash
npm run typecheck
npx vitest run tests/renderer/components/settingsContentI18nRatchet.test.ts
```

完成一个文件后，把该文件相对路径（相对 `src/renderer/components/features/settings/`）
加进 `tests/renderer/components/settingsContentI18nRatchet.test.ts` 的 `MIGRATED` 数组——
棘轮测试会扫描该文件必须无中文字面量。

## 完工报告格式（最后输出）

- 每个文件：新增键数量、复用键数量
- 未迁移的残留字符串（文件:行号 + 原因）
- 自验命令的结果
