# Code Agent 会话产品功能优化总结

## 说明

这份文档整理的是另一条独立分析线：`会话产品功能本身优化`。

它和“评测语境下的会话日志/回放/trace”不同，重点不是评测闭环，而是：

- 历史会话管理
- 单会话 trace 浏览
- turn 组织方式
- 消息/节点类型区分
- 会话列表与会话切换体验

本总结来自历史会话讨论和当时的开发上下文恢复。

## 核心背景

当时讨论的核心问题是：当前网页/客户端的 trace 展示方式，把多个会话混在同一条长流里，不符合真实使用场景。

用户明确要求：

- “不是要分割线啊，我想要的是有个历史会话列表，然后点击某个会话，查看这个会话的完整trace”
- “不可能把多个会话混在一起的”

这说明优化方向已经非常明确：

- 不是继续在单列表里加 divider
- 而是重构为真正的 `sidebar + main` 双栏会话视图

## 当时确认的核心需求

### 1. 历史会话列表

需要一个真正的会话侧边栏，而不是把多个 session 线性堆在主区里。

侧边栏需要承载的信息：

- 会话标题
- 开始时间
- 会话状态
- 请求数/turn 数
- 当前是否正在进行

用户操作方式：

- 点击某个会话
- 主区域只展示这个会话的完整 trace

### 2. 单会话完整 trace 浏览

主区域只应该展示当前选中的会话。

不应该出现：

- 多个会话混排
- 靠 session divider 做弱分隔
- 用户在长列表中自己识别边界

正确目标：

- 一个会话 = 一个完整 trace 视图
- 会话切换 = 主区域整体切换

### 3. Turn 级展开/收起

用户明确提出：

- 每一个 turn 都需要支持展开/收起

这意味着 trace 浏览器不能只做静态平铺，而应具备：

- 默认摘要视图
- 展开后看完整内容
- 支持快速扫视与深挖切换

### 4. 节点类型区分

用户关注的不是“消息都显示出来”就够了，而是要能识别每一轮/每个节点到底是什么类型。

明确提到需要区分：

- 工具调用
- 输入建议
- 模型调用
- 普通 chat 节点

并且当时已经暴露出一个具体问题：

- “工具类型标签展示的是个 `?`”

这说明现有节点类型映射和 UI label 存在问题。

### 5. 输入建议节点识别

用户特别指出：

- “这个节点，不是chat，而是用户输入建议，能区分出这个类型吗？”

这意味着消息模型不能只分：

- user
- assistant
- tool

还需要识别更细粒度的系统内节点语义，例如：

- input suggestion
- model invocation
- tool execution
- narration/status

### 6. 基础会话导航和命名能力

用户还要求了这些产品层细节：

- 左侧 `?` 按钮改为 `help`
- 页面能展示会话标题
- 页面能切换会话

这些需求说明会话页已经不只是一个 debug dump 页面，而是在往正式产品界面演进。

## 当时确认的重构方向

历史上下文里已经明确有一版结构性方案：

- 从“单滚动流 + session divider”
- 改成“sidebar + main 双栏布局”

### 已确定的结构

外层布局：

- `sidebar`
- `main`

含义：

- `sidebar`：历史会话列表
- `main`：当前选中会话的 trace 内容

### 计划中的状态模型

当时的实现方向不是继续沿用旧 divider 逻辑，而是引入真正的 session group 管理。

计划中的状态结构包括：

- `groups`
- `activeGroup`

每个 group 的结构大意为：

- `id`
- `title`
- `startTime`
- `reqIds`
- `done`

这比“在单流里插分割线”更接近真正的会话产品模型。

## 当时仍待完成的实现点

根据历史总结，当时已经进入 UI 重构中段，但尚未完全收尾。

待做项包括：

### 1. 删除旧的 divider 逻辑

待移除：

- `renderSessionDivider`
- `toggleSession`

原因：

- 旧逻辑建立在“单流中插入 session 分隔符”的思路上
- 已经不符合新的会话视图目标

### 2. 引入真正的会话分组状态

需要将：

- `sessionGroups`

重写为更正式的：

- `groups`
- `activeGroup`

### 3. 渲染侧边栏项

需要新增类似：

- `renderSessItem(group)`

职责：

- 在 sidebar 里渲染会话项
- 展示标题、时间、状态、请求数

### 4. 实现会话切换

需要新增类似：

- `selectSession(groupId)`

职责：

- 点击 sidebar item 后切换 active session
- 主区只渲染对应会话内容

### 5. 只渲染当前会话内容

需要改造主内容渲染逻辑：

- `renderCard()`

目标：

- 只渲染 active session 下的 cards/requests
- 不再混合渲染所有历史会话

### 6. 实时更新处理

新架构下还需要处理 streaming 场景：

- 新 request 属于当前 active session 时，主区实时追加
- 新 session 出现时，是否自动选中，需要有明确策略
- clear 行为需要重置 sidebar 和主区，而不是只清主区

## 这条优化线的本质

这条线本质上不是“日志优化”，也不是“评测优化”。

它是纯粹的 `session UX / conversation browsing` 优化，核心目标是：

- 让用户把会话当成一条条可管理的工作单元
- 而不是把所有请求和 trace 当成一个无限滚动日志流

这对产品体验的意义很大：

- 降低 trace 阅读负担
- 让会话切换变成显式操作
- 为后续会话搜索、会话导出、会话恢复、会话标签等能力打基础

## 推荐的产品化优先级

### P0

- sidebar 历史会话列表
- active session 切换
- 主区只展示当前会话
- turn 展开/收起
- 节点类型标签修正

### P1

- 会话标题生成/编辑
- 会话状态展示（live/done/error）
- 会话搜索和筛选
- clear/reset 与 session model 对齐

### P2

- 会话分叉
- 会话导出
- 多模型/多来源会话聚合视图
- 输入建议节点的更精细交互

## 和评测优化的边界

这条线和评测优化的关系是：

- 它们可以共享 trace 数据
- 但产品目标不同

会话产品优化关注：

- 浏览体验
- 管理体验
- 节点可读性
- 会话切换

评测优化关注：

- run/case/trace/review
- grader
- dataset
- 回归闭环

因此不应该把两者混成同一份总结。

## 结论

历史讨论里，“会话功能优化”真正指向的是：

- 从单条长流转向多会话管理
- 从弱分隔转向显式会话列表
- 从消息平铺转向结构化 trace 浏览
- 从模糊节点转向明确类型标识

如果后续继续推进这条线，正确的方向不是再补几个样式细节，而是把会话页正式升级成：

- `会话侧边栏`
- `单会话 trace 主视图`
- `可折叠 turn`
- `明确的节点类型系统`

这才是当时那轮讨论真正要解决的问题。
