# Activity Providers Migration Notes

This document is the migration slice for unifying screen activity providers. It does not describe a new runtime that already exists; it names the ownership boundaries we should converge on from the current implementation.

## Current Provider Ownership

OpenChronicle is an independent daemon provider. Code Agent does not host its collector or memory pipeline. The current owner is `src/main/services/external/openchronicleSupervisor.ts`, which resolves an external `openchronicle` CLI, runs `openchronicle start` / `openchronicle stop`, probes `http://127.0.0.1:8742/mcp`, and dynamically registers an MCP server named `openchronicle` (`src/main/services/external/openchronicleSupervisor.ts:29`, `src/main/services/external/openchronicleSupervisor.ts:81`, `src/main/services/external/openchronicleSupervisor.ts:98`, `src/main/services/external/openchronicleSupervisor.ts:210`, `src/main/services/external/openchronicleSupervisor.ts:230`). The renderer settings surface is `src/renderer/components/features/settings/tabs/OpenchronicleSettings.tsx`, which exposes daemon status, MCP health, auto injection, and blacklist settings (`src/renderer/components/features/settings/tabs/OpenchronicleSettings.tsx:35`, `src/renderer/components/features/settings/tabs/OpenchronicleSettings.tsx:54`, `src/renderer/components/features/settings/tabs/OpenchronicleSettings.tsx:181`, `src/renderer/components/features/settings/tabs/OpenchronicleSettings.tsx:200`).

Tauri Native Desktop is the bundled provider. Its collector is shipped with the app through Tauri commands and app-owned storage under `native-desktop`. The renderer bridge calls `desktop_get_capabilities`, `desktop_get_collector_status`, `desktop_start_collector`, `desktop_stop_collector`, and `desktop_list_recent_events` through `window.__TAURI_INTERNALS__` (`src/renderer/services/nativeDesktop.ts:178`, `src/renderer/services/nativeDesktop.ts:186`, `src/renderer/services/nativeDesktop.ts:190`, `src/renderer/services/nativeDesktop.ts:208`, `src/renderer/services/nativeDesktop.ts:216`, `src/renderer/services/nativeDesktop.ts:222`, `src/renderer/services/nativeDesktop.ts:226`). The Rust implementation owns the long-running collector loop, screenshot capture, dedupe, retention, SQLite/event-file persistence, and screenshot backoff (`src-tauri/src/native_desktop.rs:1438`, `src-tauri/src/native_desktop.rs:1457`, `src-tauri/src/native_desktop.rs:1471`, `src-tauri/src/native_desktop.rs:1512`, `src-tauri/src/native_desktop.rs:1601`, `src-tauri/src/native_desktop.rs:1663`, `src-tauri/src/native_desktop.rs:1718`).

The bundled provider already has Node-side read and processing services. `NativeDesktopService` resolves the app-owned `native-desktop` root and exposes status, current context, recent events, timeline, search, and stats (`src/main/services/desktop/nativeDesktopService.ts:36`, `src/main/services/desktop/nativeDesktopService.ts:53`, `src/main/services/desktop/nativeDesktopService.ts:189`, `src/main/services/desktop/nativeDesktopService.ts:219`, `src/main/services/desktop/nativeDesktopService.ts:223`, `src/main/services/desktop/nativeDesktopService.ts:227`). Derived processing lives in `src/main/desktop/desktopActivityUnderstandingService.ts`, which turns raw events into summaries, todo candidates, and semantic matches (`src/main/desktop/desktopActivityUnderstandingService.ts:1`, `src/main/desktop/desktopActivityUnderstandingService.ts:4`, `src/main/desktop/desktopActivityUnderstandingService.ts:841`, `src/main/desktop/desktopActivityUnderstandingService.ts:852`). That processing is the part most suitable for future sidecar extraction because it is not the Tauri capture primitive itself; it reads persisted activity and writes derived artifacts.

Audio is part of the bundled desktop activity surface but should remain explicitly user-visible. The current audio service records through `sox rec`, VAD, and ASR (`src/main/services/desktop/desktopAudioCapture.ts:1`, `src/main/services/desktop/desktopAudioCapture.ts:20`). The UI keeps audio as a manual button and mode selector in the desktop panel (`src/renderer/components/features/settings/sections/NativeDesktopSection.tsx:1035`, `src/renderer/components/features/settings/sections/NativeDesktopSection.tsx:1050`), and the IPC layer documents why it is not auto-started by meeting-app detection (`src/main/ipc/desktop.ipc.ts:20`).

## Target Interfaces

`ActivityProvider` is the source adapter. It answers: what provider is installed, whether it is running, what permissions or daemon health are blocking it, and how to query activity records. OpenChronicle and Tauri Native Desktop are two implementations with different lifecycle ownership. OpenChronicle provider delegates lifecycle to an external daemon and reads through MCP. Tauri provider owns bundled capture and reads local app data.

`ActivityContextProvider` is the normalized context builder. It should consume one or more `ActivityProvider` implementations and return a bounded, redacted context object for the agent. The current OpenChronicle path already does a narrow version of this in `fetchOpenchronicleContext()`: it loads OpenChronicle settings, calls the MCP `current_context` tool, filters captures, formats headlines/timeline, and returns a short text block (`src/main/services/external/openchronicleContextProvider.ts:197`, `src/main/services/external/openchronicleContextProvider.ts:204`, `src/main/services/external/openchronicleContextProvider.ts:206`, `src/main/services/external/openchronicleContextProvider.ts:213`). Native desktop has raw and derived data paths but no equivalent single provider facade yet.

`ContextInjectionPolicy` decides when and how activity context enters the prompt. It should not be owned by either provider. Today injection is provider-specific: `conversationRuntime` calls `fetchOpenchronicleContext()` and injects `<screen-memory>` directly at session start (`src/main/agent/runtime/conversationRuntime.ts:804`, `src/main/agent/runtime/conversationRuntime.ts:806`, `src/main/agent/runtime/conversationRuntime.ts:808`). The older contextual memory hook is a no-op (`src/main/agent/runtime/streamHandler.ts:53`), and native-derived activity is bootstrapped separately for non-simple tasks (`src/main/agent/runtime/conversationRuntime.ts:815`). The target is to move the decision into a policy layer that can choose provider, token budget, redaction level, timing, and disable reasons.

The frontend should reflect the same split. The global desktop panel already renders the bundled provider timeline (`src/renderer/App.tsx:546`, `src/renderer/App.tsx:557`) through `NativeDesktopSection`, which loads collector status, recent events, and audio status (`src/renderer/components/features/settings/sections/NativeDesktopSection.tsx:896`, `src/renderer/components/features/settings/sections/NativeDesktopSection.tsx:908`). The settings modal currently exposes only the OpenChronicle settings tab for screen memory (`src/renderer/components/features/settings/SettingsModal.tsx:85`, `src/renderer/components/features/settings/SettingsModal.tsx:147`). Long term, the user-facing IA should be one "Activity / Screen Memory" surface with provider cards plus one timeline, not separate mental models.

## Migration Phases

### Phase 1: ActivityContextProvider

Create a provider-neutral `ActivityContextProvider` boundary in main-process code. The first version should wrap the existing OpenChronicle context path without changing behavior, then add a Tauri Native Desktop implementation that can return a small recent-context payload from `NativeDesktopService.getTimeline()` and/or derived summaries. The important migration step is naming the normalized output and source metadata: provider id, provider type, captured window/app/url fields, redaction status, freshness, and confidence.

This phase should also make provider availability explicit. OpenChronicle availability is daemon/MCP health from `openchronicleSupervisor` and `OpenchronicleStatus`; Tauri availability is Tauri runtime plus collector status and TCC permissions from `nativeDesktop.ts` / `native_desktop.rs`. The current renderer already distinguishes Tauri availability through `window.__TAURI_INTERNALS__` (`src/renderer/services/nativeDesktop.ts:178`), but the prompt path does not yet consume that as a provider capability.

### Phase 2: Prompt Injection Switch

Move prompt injection from direct OpenChronicle calls to `ContextInjectionPolicy`. The current direct injection is hard-coded in `conversationRuntime` (`src/main/agent/runtime/conversationRuntime.ts:804`). The policy should decide:

- whether activity context is allowed for this session;
- which provider wins when both OpenChronicle and Tauri Native Desktop are enabled;
- how many tokens can be injected;
- whether only summaries are allowed or raw titles/URLs can appear;
- whether injection happens only at session start or can be refreshed later;
- what audit metadata is recorded for injected context.

Default behavior should preserve today's OpenChronicle injection when the OpenChronicle toggle and `autoInjectContext` are enabled (`src/shared/contract/openchronicle.ts:5`, `src/shared/contract/openchronicle.ts:12`). The migration should then allow switching to bundled native context without duplicating `<screen-memory>` blocks or mixing two providers' raw captures into one untraceable prompt.

### Phase 3: Tauri Native Long-Termization / Sidecar Evaluation

Keep capture bundled in Tauri until there is a clear reason to split it. The Rust collector is tied to Tauri commands, app storage, macOS screen capture, and TCC-sensitive behavior (`src-tauri/src/native_desktop.rs:1457`, `src-tauri/src/native_desktop.rs:1601`). Moving that too early would create install, permission, and lifecycle complexity without reducing much product risk.

Evaluate sidecar extraction for heavy or long-running processing first: visual analysis, audio/ASR processing, and desktop activity understanding. These already sit outside the Rust capture loop or read persisted data from the native desktop store (`src/main/services/desktop/desktopVisionAnalyzer.ts:164`, `src/main/services/desktop/desktopAudioCapture.ts:401`, `src/main/desktop/desktopActivityUnderstandingService.ts:852`). A sidecar can later own queueing, backpressure, model selection, and crash isolation while the Tauri provider remains the bundled source of raw activity.

## Risks To Keep Explicit

- Privacy leakage: app names, window titles, browser URLs, document paths, screenshots, transcripts, and derived summaries can reveal sensitive work. OpenChronicle has blacklist filtering before injection (`src/main/services/external/openchronicleContextFilter.ts:45`), while Tauri native collector has `redactSensitiveContexts` and screenshot suppression for sensitive contexts (`src-tauri/src/native_desktop.rs:1467`, `src-tauri/src/native_desktop.rs:1565`, `src-tauri/src/native_desktop.rs:1601`). These policies must stay visible in UI and injection metadata.
- TCC permissions: screen recording, accessibility, microphone, and system audio permissions can fail independently. The Tauri bridge exposes permission/status/system-settings actions (`src/renderer/services/nativeDesktop.ts:190`, `src/renderer/services/nativeDesktop.ts:194`, `src/renderer/services/nativeDesktop.ts:321`), and the workbench hook already treats readiness as a first-class state (`src/renderer/hooks/useWorkbenchBrowserSession.ts:259`, `src/renderer/hooks/useWorkbenchBrowserSession.ts:351`).
- Daemon lifecycle: OpenChronicle keeps running outside Code Agent until the user turns it off (`src/main/services/external/openchronicleSupervisor.ts:4`, `src/main/services/external/openchronicleSupervisor.ts:10`). That is correct for an independent provider, but the UI and policy must not imply app-close stops collection.
- Audio提示: audio capture must remain manual and visible. The code already avoids automatic meeting-app startup because macOS labels ScreenCaptureKit as screen sharing and the user experience is misleading (`src/main/ipc/desktop.ipc.ts:20`). Any provider abstraction should preserve that guardrail.
- Prompt injection pollution: unbounded activity context can distract or bias the model. Today OpenChronicle injection is capped and formatted (`src/main/services/external/openchronicleContextProvider.ts:16`, `src/main/services/external/openchronicleContextProvider.ts:17`, `src/main/services/external/openchronicleContextProvider.ts:177`), but native-derived context needs the same budget and source labelling before it can be injected.
- Automated action misfire: desktop activity can influence browser/computer-use readiness and suggested actions. The policy must separate "context for understanding" from "permission to act". Current desktop IPC exposes observation, element listing, managed browser session state, and audio controls under the same desktop domain (`src/main/ipc/desktop.ipc.ts:35`, `src/main/ipc/desktop.ipc.ts:38`, `src/main/ipc/desktop.ipc.ts:74`, `src/main/ipc/desktop.ipc.ts:92`, `src/main/ipc/desktop.ipc.ts:166`), so automatic action gating must remain outside the context provider.

## Documentation Decision

Use "provider" for source ownership and lifecycle. Use "context provider" for the normalized prompt-ready data builder. Use "injection policy" for the final prompt decision. Do not let OpenChronicle's current `<screen-memory>` path become the generic architecture by accident, and do not treat the Tauri native collector as just another external daemon. It is bundled today; only its heavier processing should be evaluated for sidecarization first.
