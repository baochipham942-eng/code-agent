## Deferred（不在本轮修，带理由）

**🔴 IPC 会话授权 fail-open / callerSessionId 可伪造**（workflow.ipc.ts approve/reject）
- Codex 观点：`callerSessionId` 由 renderer payload 自报、可选、缺省放行，可伪造另一会话 sessionId 通过；应从 `_event` 连接上下文取可信 session。
- Deferral 理由：Neo 是**单用户本地桌面 app**（Tauri）+ localhost webServer（共享 token 注入到服务的 HTML，多 tab 共享同一 token）——**IPC 层不存在 per-连接/per-会话身份**，Codex 建议的「从 _event 取可信 session」在此架构下不可实现。renderer 是用户自己的可信客户端；自报 callerSessionId 已挡住**正常使用的误批**（真实 UX 风险）；伪造 = 用户对自己的数据构造 IPC 调用，无提权、无新增能力。兄弟件 swarm approve/reject **零会话校验**，workflow 已更严。
- **Codex Round 4 终审认可**：「在你写明的 threat model 下是合理的，不算 must-fix……这层最多只能做防误触的 UX guard，不是安全边界。别再把它叫 auth。」→ 已按此把它定位为 UX guard 而非 auth boundary。
- 翻盘条件：若 Neo 演进到多用户/远程/不可信 renderer，或 IPC 层引入可信 per-session 身份（如 main 端签发一次性 request capability），则升级为 must-fix。
