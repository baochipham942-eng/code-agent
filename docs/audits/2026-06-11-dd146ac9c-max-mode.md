## Findings by Round

### Round 1（发散 8 维度）

#### 🔴 HIGH — judge 裁决可被 spoof（parseJudgeWinner last-match-anywhere）
**Finding**: `parseJudgeWinner()` 取全文最后一个 `WINNER:` 匹配，judge 行文引用候选注入的 `"WINNER: 1"` 字样即可劫持裁决——赢家的 tool call 会被下游真实执行，这是攻击面。
**Resolution**: ✅ fixed in 16e518a2d — 只认最后非空行的整行锚定 WINNER 行；新增注入/尾随文本/空白测试。
**Why**: 候选内容是 attacker-controlled（提示注入场景），judge 提示词无法保证格式纪律。

#### 🔴 HIGH — 取消可泄漏部分赢家进主链路
**Finding**: 候选错误（含 abort）一律吞成 null。取消时已完成的候选会成为幸存者被 replay，其 tool call 流入下游持久化/执行。
**Resolution**: ✅ fixed in 16e518a2d — `MaxModeDeps.isAborted` 探针在候选/judge 完成后检查，命中即抛错走外层既有取消语义（空文本响应）。R2 进一步补了沉没成本记账（见 R2-M1）。

#### 🔴 HIGH — 预算闸被 N 并发绕过
**Finding**: 预算检查在 step 前只做一次（advisory），Max Mode 在临界状态一步并发花出 N+1 笔调用，事后记账拦不住。
**Resolution**: ✅ fixed in 16e518a2d — `maxModeBudgetHeadroomOk()`：alertLevel 为 WARNING/BLOCKED 时跳过扇出，退回正常单次流式调用。

#### 🟡 MEDIUM — 静默候选仍发 model_decision 事件
**Finding**: 每个候选+judge 都走 `resolveMainChatModelDecision()` 发 `model_decision` 事件，N+1 条决策事件混进 UI/遥测，与真实赢家不可区分。
**Resolution**: ✅ fixed in 16e518a2d — `InferenceOptions.suppressModelDecisionEvent`，静默调用只抑制事件、路由行为不变。

#### 🟡 MEDIUM — overhead 按请求模型计价（adaptive 路由后错账）
**Finding**: `recordTokenUsage()` 固定记 `ctx.modelConfig`，adaptive 路由到免费模型的候选会按昂贵的请求模型计价（或反之）。
**Resolution**: ✅ fixed in 16e518a2d — `stats.overheadEntries` 逐条携带 `actualProvider/actualModel`，wiring 按实际模型 `budgetService.recordUsage`。

#### 🟡 MEDIUM — 候选数无界且非整数不安全
**Finding**: `Math.max(1, candidates)` 不拦 NaN/Infinity/10000，错误配置可触发海量并发扇出。
**Resolution**: ✅ fixed in 16e518a2d — `normalizeCandidateCount()`：非安全整数回落 1，硬上限 `MAX_MODE.MAX_CANDIDATES=10`。

#### 🟢 LOW — maxMode 诊断缺共享事件契约类型
**Resolution**: ✅ fixed in 16e518a2d（顺手修）— `shared/contract/agent.ts` 的 `model_response.runtimeDiagnostics` 补 `maxMode` 形状。

### Round 2（聚焦回归/完整性/对称应用）

#### 🟡 MEDIUM — R1-H2 的修复丢了取消步的沉没成本
**Finding**: isAborted 直接抛错后，已完成候选/judge 的真实 token 消耗不进 `recordUsage`，预算被低估。Codex caught a real issue I missed——这是 R1 修复引入的回归类 finding。
**Resolution**: ✅ fixed in dd146ac9c — `MaxModeAbortError` 携带 `overheadEntries`，wiring 重抛前按实际模型记账。

#### 🟡 MEDIUM — R1-H1 的锚定解析收得过死
**Finding**: judge 在裁决行外套代码 fence 或补个句号就 fail-open 回候选 0，把 judge 已选掉的更优候选换回去。
**Resolution**: ✅ fixed in dd146ac9c — 倒序跳过尾部空行/纯 fence 行再锚定，容忍单个尾随标点。刻意不做全文 fence 过滤（会重新暴露"裁决后 fenced 注入"）。

#### 🟢 LOW — 赢家/降级调用仍按请求模型记账
**Finding**: M2 只对 losers/judge 做了实际模型分账，赢家估算（inference.ts 主路径）和降级单次调用仍走 `ctx.recordTokenUsage` 按请求模型记。
**Resolution**: ❌ deferred — 这是主链路既有记账语义（开关关时同样存在，影响所有正常 inference 的 fallback 场景），只改 max-mode 侧会造成与正常路径的不对称。应作为独立的主链路记账改进单独立项。

**Sibling note（Codex 主动豁免）**: `goalReviewGate.parseVerdict` 仍是 last-match-anywhere，但 fail-closed 且不吃 attacker-controlled 候选内容，threat model 不同，不算本分支漏改。

## Convergence Analysis

两类教训值得记：
1. **R2 的两个 MED 都是 R1 修复自身引入的**（沉没成本被 abort 丢掉、锚定解析过死）——
   "修复引入的回归"再次成为 Round N+1 的第一来源，印证 skill 把 regression class
   排在 Round N 检查首位的设计。修 fail-open/abort 类问题时，要同时问"丢弃的路径上
   有没有已发生的副作用（钱/状态）需要结算"。
2. **判别式 LLM 输出的解析协议是安全面**：parseJudgeWinner 的 spoof（R1-H1）与过严
   （R2-M2）是同一根因的两面——文本协议既要防注入又要容忍格式噪声。锚定 + 尾部
   fence/标点豁免是当前平衡点；后续若上结构化输出（tool_choice 强制 JSON）可整体消除。
