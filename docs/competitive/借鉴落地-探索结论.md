## 一批实现细节（已 as-built 核实）

### 1 · 预算接线修复（bug 级，最高优先）
- **现状**：`budgetService.ts` 三级告警(70/85/100%)+agent 链路接入完整，但**两断点**：① `initBudgetService(config)` 启动期从未被调用 → 单例跑硬编码 $10/24h；② `configService.setBudgetConfig`(`configService.ts:1245`) 无任何调用方。前端 `CostDisplay.tsx` 只显累计美元、不显上限/百分比/染色。
- **方案**：启动期 `initBudgetService(getBudgetConfig())` + `setBudgetConfig` 内同步单例 `updateConfig`。不接线一切预算 UI 都是假的。
- **工时**：0.5d。**文件**：`budgetService.ts`、`configService.ts:1245`。

### 2 · parse-error 回贴 schema
- **现状**：`toolExecutionEngine.ts:490-535` parse-error 分支只回截断 raw args 不贴 schema 并提前 return；validation 分支(`:541-592`)失败才注入 schema，靠 `toolArgsValidator.ts:124-148 formatValidationError`(简化 schema)。两分支 schema 同源 `getToolDefinitionWithCloudMeta(name).inputSchema` 可复用。
- **方案**：抽 `export formatSchemaForModel(properties, required)`（纯重构）；parse-error 分支取 inputSchema 调它追加（无 schema 跳过、字段 >25 只列 required）。
- **工时**：0.5-1.5h。**风险**：低（纯加法）。**文件**：`toolExecutionEngine.ts`、`toolArgsValidator.ts`。

### 3 · 压缩卡死护栏 + 无损剪枝短路（+8 防漂移钉）
- **现状（关键纠正）**：系统 A `compressionPipeline.ts` 的 L1 剪枝已含 error 保留、L4 阈值门已实现"降下去就不付费摘要"——**无损剪枝那半在系统 A 已覆盖，不改**。真风险在系统 B `contextAssembly/compression.ts:278-416 checkAndAutoCompress`（`messageProcessor.ts:1059` 每轮跑、直接付费摘要、**无连续压缩计数**）。
- **方案**：① 卡死护栏：`runtimeContext.ts:234` 旁加 `_consecutiveCompacts` + `MAX_CONSECUTIVE_COMPACTS=2`，压后仍≥阈值则++/降下去清零，≥2 注入 system message 暂停自动压缩（仿 `shouldWrapUp`）；② 剪枝短路：系统 B 付费摘要前先跑 `applyToolResultBudget`(留 error) 重测阈值，降下去跳过摘要；**必须把活跃文件最后一次 Read 加进 protected**（搬 `autoCompressor.ts:148-186`），阈值判降同口径(currentTokens vs triggerTokens)。③ **#8 防漂移钉**：确认压缩保留"用户原话"逐字不被二次摘要（参照 Reasonix `compact.go:350-369`）。
- **工时**：1-1.5d，两改+防漂移钉同 PR。provider 无关。**文件**：`compression.ts`、`runtimeContext.ts`、`context/layers/toolResultBudget.ts`、`autoCompressor.ts`、`contextPressureController.ts`。

### 4 · 🔴 发送前统一悬空 tool_call 修复 + 半截 args 闭合
- **背景**：Reasonix `SanitizeToolPairing`(`provider.go:97-127`) 与 Codex `normalize_history` **独立收敛到同一范式**——在**每次组装 prompt 前**(覆盖 resume/fork/abort 全路径)修：未答 tool_call 回填占位结果、孤儿 tool 消息丢弃、半截流式 args JSON 自动闭合(DeepSeek 重放半截 args 会 400)。
- **neo 缺口**：单会话在飞工具靠账本去重，但无"发送前统一配对修复"。neo 多 provider 桌面 + resume 是核心交互，配对 400=任务断裂。即 `reasonix-借鉴清单.md` P2-4，对照确认升档。
- **方案**：在发模型前的历史组装处（`contextAssembly/inference.ts` 附近）加一道 sanitize：扫 assistant tool_calls 无对应 tool 结果→回填占位；孤儿 tool 消息丢弃；末条半截 args 闭合。存储保原文，只在发送时修。
- **工时**：中(~60 行) + 测试。**先核**：确认 neo 现在到底有没有等价 sanitize（grep toolPairing/orphan/danglingToolCall），别重做。

### 5 · 🔴 成功写 storm breaker
- **背景**：Reasonix `repeatSuccessBreakThreshold=2`(`agent.go:1537-1584`)：模型反复做同一**成功**写(没 error 给 doomLoop 看)，按规范化写签名满 2 次即 block 提示换 edit/verify。
- **neo 缺口**：`doomLoopGuard.ts` 只防失败循环/同调用循环，不防"反复成功写同一文件"的隐性空转。
- **方案**：doomLoopGuard 加一类"成功写签名"分支（write_file/edit_file/multi_edit + 会写文件的 bash），同写满 2 次触发 nudge。复用现有签名/计数框架。
- **工时**：低-中。**文件**：`doomLoopGuard.ts`。

### 9 · ⭐ 长输出留头尾弃中间 + 进度条压缩
- **现状**：`ToolCallDisplay/index.tsx` BashOutputPreview `slice(0,20)` 首 20 行硬截断 → build/test 末尾结果被砍。
- **方案**：改头 20% + 尾 80% + `[…N 行省略]` 中间标记；对 `\r`/`\b` 刷屏的 spinner/进度条折叠（学 Roo）。
- **工时**：0.5-1d。**文件**：`ToolCallDisplay/index.tsx`(BashOutputPreview)、`ToolDetails.tsx`。

### 10 · ⭐ 错误一键重试/复制 + 喂回 agent
- **现状**：`toolExecutionPresentation.ts:137 getToolRecoveryHint` 只是 hint 文字"可重试"；错误卡有"查看原始报错"折叠 + `humanizeToolError` 人话(已好)，但无通用 copy 按钮、无可点重试。
- **方案**：给文本/报错 `<pre>` 加 hover 复制；把"重试这步/把报错喂回 agent"做成可点 action，复用已有 `BrowserComputerNextStepActions` executable-action 基建泛化到通用工具。
- **工时**：0.5-1d。**文件**：`ToolDetails.tsx`、`toolExecutionPresentation.ts`。

### 留待 roadmap：#15 provider 限流剩余位（完整侵入版）
探索结论设想"限流剩余/limit/reset 存进 `providerHealthMonitor`"听着像加字段，**but as-built 前提不成立**：
- `recordSuccess`（`retryStrategy.ts:246`）拿到的是泛型解析结果 `T`（文本/流），**不含原始 HTTP header**——成功路径拿不到 `x-ratelimit-*`。
- header 目前只在**错误路径**可达（`errorClassifier` 读 `x-ratelimit-remaining`、`retryStrategy` 抽 `retry-after`，皆 `err.headers`）。
- 各 provider 响应解析各异（`anthropic-ratelimit-*` vs `x-ratelimit-*` vs 无）。

要做**连续剩余额度**，须改**每个 provider adapter** 把 raw response header 线出来汇到公共路径（`withTransientRetry` 成功分支或各 provider 解析处）→ 存进 `providerHealthMonitor` → 经 `provider.ipc getHealthStatus` 暴露给 StatusBar 常驻位，缺失显"未知"。**估 ~1.5-2d，跨 provider 侵入，需逐个验证 header 命名**。

轻量替代（未采纳）：仅从 429/限流错误捕获 {remaining/limit/reset/retry-after}（这条路 header 已可达）——但只是**事后**信号（被限流后才有数），价值偏薄。

→ 决策：完整版留待 roadmap；轻量版价值不足不单独做。重启点 = "改每个 provider adapter 线出 response header"。
