# ADR-022: Append-Only 事件账本 + Canonical Event Spine

> 状态: accepted（已拍板 2026-06-16）

## 一、背景与问题：为什么"多本账"是隐患

把 Code Agent 想象成一家公司。公司里发生的每件事——用户说了句话、模型回了句话、调用了工具、做了一次权限判断、几个 Agent 协同跑了一个任务——都应该被记录下来，事后能查、能复盘、能在崩溃后恢复现场。

**今天的问题是：这家公司有 5 套互不相通的账本，各记各的，没有一本"总账"。**

| 记的是什么事 | 记在哪本账 | 能不能改 | 出事了能不能恢复 | 记得严不严谨 |
|---|---|---|---|---|
| 用户和模型的对话 | 消息表 | 能改（可编辑、可撤销） | 靠"同步标记"，不重放 | 严谨 |
| 每一轮的成本/质量遥测 | 遥测表簇 | 整轮覆盖重写 | 失败时打个脱敏包待上传 | 松散（塞在一坨文本里） |
| 多 Agent 协同的轨迹 | Swarm 表 + 一份 JSONL 文件 | 运行记录会被覆盖 | 靠 JSONL 文件半行自愈 | 中等 |
| 任务状态变化 | 任务事件表 | 只追加 | 没有专门的恢复机制 | 松散（类型靠应用层口头约定） |
| 权限决策的来龙去脉 | 只在评测时存一下 | 只读 | 产品正常跑时根本不记 | 严谨但几乎不落地 |

**这套"分账"模式带来三个真实的业务风险：**

1. **复盘难、说不清。** 用户投诉"刚才这步为什么被拒了/为什么花了这么多钱",我们得同时翻 3-4 本账，还经常对不上号。决策链（为什么允许/拒绝某个操作）在产品正常运行时甚至**根本没记**，只有跑评测才存——等于现场没有监控录像。

2. **崩溃恢复弱。** 程序崩了重启，现在只能把会话状态从"运行中"打个标记改成"已中断",**没法把崩溃前那一刻正在做的事完整重放回来**。用户看到的是一句"建议恢复",而不是真的接着干。

3. **越改越乱，加一个功能要改五个地方。** 每新增一类要追踪的事件,就要在 5 套账本里挑一套塞进去,或者再开第六套。长期下去维护成本只增不减,数据一致性全靠"大家自觉"。

**这个 ADR 要决策的核心是：要不要立一本"总账"(canonical event spine,可以理解成公司的中央流水账),让所有事件先进总账、再分发到各个明细表;以及这本总账用什么形式来存。** 总账有一个关键特性叫 **append-only(只追加、不修改)**——像银行流水,只能往后记新的一笔,不能涂改旧的,这正是"可审计、可重放、可恢复"的基础。

## 二、待决策点

### 决策点 1（核心）：总账用什么形式存？

这是真正需要您拍板的一题。它决定了"总账"这本流水账物理上放在哪里、和现有数据库是什么关系。三个选项：

#### 选项 A：事件流单独成文件，与数据库物理隔离（照 maka 原设计）

> 通俗讲：流水账记在一沓**独立的纸质账本(JSONL 文件)**里,和数据库这台"电脑账"完全分开放。数据库继续管"当前状态"(谁在线、任务到哪步了),文件账本管"历史上发生过什么"。

- **优点**
  - 职责最干净:文件只管"发生过的事",数据库只管"现在的状态",两边互不污染。
  - 事件流是天然 append-only,文件追加写最简单、最不容易写错。
  - 已有先例:现在 Swarm 轨迹就是这么干的(SQLite + JSONL 双写),团队踩过坑、有经验。
  - 出事时文件账本是独立证据,数据库挂了不影响这沓"纸"。
- **缺点**
  - **两套存储要保持一致很难**——这就是下文要讲的"崩溃瞬间双写不一致"陷阱的高发区。
  - 想做"既要查当前状态、又要关联历史事件"的复杂查询时,要跨"文件"和"数据库"两个世界拼接,麻烦。
  - 文件会越积越多,要单独做清理、归档、容量管理。

#### 选项 B：事件表放进数据库,和明细表同步双写

> 通俗讲:总账也记在数据库里,新开一张"事件表"。每发生一件事,既往总账(事件表)记一笔,又往对应明细表(消息表/任务表…)写一笔,**两笔一起提交**。

- **优点**
  - 全在一个数据库里,跨表关联查询最顺手(查对话时顺带把它对应的事件、成本、决策全捞出来)。
  - 数据库自带事务保证,"两笔一起成功或一起失败",一致性比选项 A 强。
  - 不用额外管理文件,备份/同步/清理都复用现有数据库那一套。
- **缺点**
  - 总账和明细表**职责会粘在一起**,长期容易退化回"分账"的老问题。
  - 数据库体积增长更快(同一份信息事实上存了两遍)。
  - "双写"如果没用好数据库事务,同样会踩一致性陷阱(见下文)。

#### 选项 C：混合——关系查询走数据库,只把"事件流 + 大块产物"外置

> 通俗讲:**该用数据库的用数据库,该用文件的用文件。** 需要被频繁关联查询的(对话、任务状态、成本汇总)留在数据库;纯粹的"历史事件流水"和"大块产物"(比如一次完整的 Agent 运行轨迹、生成的大文件)外置成文件。两边用一个"事件 ID"挂钩。

- **优点**
  - 各取所长:查询性能和审计/重放能力都照顾到。
  - 数据库不会被海量历史事件和大块产物撑爆,保持轻快。
  - 演进灵活:可以先把最痛的一类(比如决策链、任务事件)纳入总账,其余渐进迁移。
- **缺点**
  - **架构最复杂**,要同时维护"数据库 + 文件 + 两者的挂钩关系",心智负担最高。
  - 一致性保证介于 A 和 B 之间,需要一套后台校验来兜底(定期对账,发现对不上就修)。
  - 对"什么放数据库、什么外置"的边界判断要求高,划错了会反复返工。

**关于"崩溃瞬间双写不一致"这个陷阱属于哪个选项——这是您拍板前必须知道的一点：**

> **这个陷阱主要属于选项 A 和选项 C,在选项 B 里最轻。**
>
> 陷阱长这样:程序要把"一件事"同时记到两个地方(选项 A 是"文件 + 数据库",选项 C 是"数据库内 + 外置文件")。假如刚写完第一处、还没写第二处时程序崩了,两边就**对不上账**了——一边说这事发生了,另一边查无此事。
>
> - **选项 B 几乎没有这个陷阱**:因为它两笔都写在同一个数据库里,可以用数据库的"事务"把两笔捆成一个原子操作——**要么都成功,要么都不算**,不存在写一半的中间态。
> - **选项 A / C 有这个陷阱**:文件系统和数据库是两个独立世界,没有跨它们的统一事务。要么接受"最终一致"(崩溃后靠重放/对账修复,短时间内可能不一致),要么自己实现一套"先写总账、再补明细"的恢复协议。这是选 A/C 必须额外买单的工程成本。

---

### 决策点 2：先全量改造,还是先改一类试水?

无论选 A/B/C,都要决定改造节奏:

- **选项 X:大爆炸式全量。** 一次性把 5 套账本全收进总账。**优点**:一步到位,不留历史包袱。**缺点**:改动面巨大、风险高、周期长,中途出问题难定位。**不推荐。**
- **选项 Y:最小切口先行(推荐)。** 先挑一个最痛、最干净的场景接入总账(建议先做"权限决策链"——因为它现在产品态根本没记,属于纯增量、不破坏现有任何东西),跑通"先进总账、再分发"的闭环,验证恢复/重放真的有用,再逐类迁移。**优点**:风险可控、能快速拿到证据、随时可停。**缺点**:过渡期内"半新半旧"会并存一段时间。

---

## 三、推荐方案

**决策点 1 推荐选项 C(混合),决策点 2 推荐选项 Y(最小切口先行)。**

理由用一句话说:**纯文件隔离(A)在我们这种"既要复盘历史、又要实时查状态"的产品里,跨世界查询和一致性都吃亏;纯数据库双写(B)虽然一致性最稳,但长期会把总账和明细重新搅在一起、还撑大数据库;混合(C)能让数据库保持轻快做查询、让事件流和大块产物外置做审计,代价是要接受一次性把架构搭复杂、并配一套后台对账来兜住一致性陷阱。**

而且我们已经在 Swarm 场景验证过"数据库 + 文件双轨"是能跑的,选 C 等于把这个被验证过的模式正规化、推广开,而不是发明新东西。

## 四、拍板后怎么分期推进

> 拍板后由 Claude 自主实现 + 分层验证,不需要您看代码。各期都以"能拿出可演示的证据"为交付标准。

- **第一期 · 立账本骨架(最小切口)**
  把"总账"这张事件表/事件流先立起来,接入**权限决策链**这一个场景(纯增量,不动现有任何表)。交付证据:能在界面或日志里看到"每一次允许/拒绝都留下了完整决策流水"。

- **第二期 · 接入崩溃恢复**
  让程序崩溃重启后,能从总账把"崩溃前正在做的事"重放回来,而不只是打个"已中断"标记。交付证据:演示一次"运行中强杀进程 → 重启 → 现场被完整恢复"。

- **第三期 · 迁移高价值场景**
  把任务事件、Swarm 轨迹逐步收进总账,旧表降级为"读优化缓存"。交付证据:复盘一个真实会话时,从一本账就能看清"对话 + 任务 + 协同 + 成本"全链路。

- **第四期 · 一致性兜底 + 老数据迁移**
  上线后台对账(定期检查总账和明细对不对得上,发现偏差自动修),并迁移历史数据库。交付证据:对账报告 + 老库平滑升级。

> 如果第一期跑下来发现收益不及预期,可以随时叫停,沉没成本仅限第一期。

## 五、影响范围

- **用户可感知的变化**:复盘/客服排障能"一本账查到底";崩溃恢复从"提示建议"升级为"真的接着干";对功能本身的日常使用**无感知、不改变操作习惯**。
- **数据层**:新增一套"事件总账"存储(表或文件,取决于选项);现有消息/遥测/Swarm/任务等表的写入路径要改造为"经总账分发"(分期进行,不是一次性)。
- **可靠性**:崩溃恢复能力显著增强;但引入新的一致性维护责任(尤其选 A/C 需要后台对账)。
- **性能与容量**:选 B 数据库增长加快;选 A/C 需新增文件清理/归档机制。
- **迁移成本**:存量本地数据库需要一次性升级脚本(放在最后一期,前面几期对老用户向后兼容)。
- **不影响**:发版流程、模型路由、计费口径、现有 UI 交互。
