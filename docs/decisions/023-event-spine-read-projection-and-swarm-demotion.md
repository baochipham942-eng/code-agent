## 一、为什么现在要补这个 ADR

ADR-022 拍板了"立一本总账（canonical event spine），让所有事件先进总账、再分发到各明细表"。但落地一/二期后，出现两个 ADR-022 没讲清、需要你确认的点：

1. **"总账"到底长什么样？** 一/二期实际上**没有建一张物理的"总账大表"**，而是按场景各建了一张只追加的小账本（权限决策一张、工具执行一张）。这其实更稳——但它偏离了 ADR-022 字面上的"单一总账表"。第三期要"把任务事件、Swarm 收进总账"，必须先说清"总账"是一张物理大表，还是一个**逻辑上的统一读出面**。

2. **"旧表降级为读优化缓存"具体怎么降？** ADR-022 §四第三期写了这句话，但没给策略。Swarm 的旧表是**会被覆盖重写的**（同一次运行重启会覆盖、运行结束会改写统计），把它"降级为缓存"等于动 Swarm 的核心写路径——这是整个事件账本工程里**风险最高的一步**，必须有明确的过渡策略，不能闷头改。

这两点定了，第三期才好分两段走：**3a（低风险，本 ADR 决策点 1）先交付"一本账复盘"招牌证据；3b（高风险，本 ADR 决策点 2）再做 Swarm 降级。**

---

## 二、决策点 1：总账用「物理大表」还是「读侧投影」？

> 通俗讲：现在有好几本小账本（决策一本、执行一本、任务一本、协同一本、成本一本），各记各的、都只追加。"复盘一个会话"时，要把它们按时间合成"一本账"给人看。问题是：这"一本账"要不要在数据库里真建一张大表把所有事件抄进去？

### 选项 P1：建物理大表（照 ADR-022 字面）
新建一张 `events` 大表，所有场景的事件都往这张表里抄一份，明细表退居二线。
- **优点**：字面忠于 ADR-022；"一本账"物理上真的是一张表，查询直观。
- **缺点**：每件事都要存两遍（大表一遍、明细表一遍），数据库膨胀快；要改所有场景的写路径往大表灌；和"增量、随时可停"的纪律冲突——这是个大改造。

### 选项 P2（推荐）：读侧逻辑投影，不建物理大表
保持"各场景各一本只追加小账本"不变，**复盘时临时把它们按时间合并成"一本账"读出来**（一个只读的"投影"服务），不落地成新表。
- **优点**：零新增写入、不动任何写路径、不膨胀数据库；"一本账"该有的复盘价值全拿到（按时间看清对话+任务+协同+成本+决策+执行）；纯只读，风险极低，随时可停。
- **缺点**：每次复盘要现场合并（性能开销在读，不在写）——但复盘是低频操作，可接受；如将来复盘量极大，可再加缓存，不影响现在的判断。

**推荐 P2。** 理由一句话：**一/二期已经用"分场景小账本"跑通了，"总账"的真正价值是"复盘时能合成一本账看"，这件事用只读投影就能完全做到，没必要为它付"建大表 + 全场景双写 + 数据库膨胀"的重成本。** ADR-022 的"spine"在本项目落地为**逻辑总账（统一读出面）**，而非物理单表——这是对 ADR-022 的细化，不是推翻。

## 三、决策点 2：Swarm「旧表降级」用什么过渡策略？（管 3b）

> 背景：Swarm（多 Agent 协同）现在的账记得最乱——既有会被覆盖重写的 SQLite 统计表（`swarm_runs`/`swarm_run_agents`），又有一份只追加的 JSONL 轨迹文件，两套并存。ADR-022 要把它"降级为读优化缓存"，意思是：让**只追加的事件流当唯一真理源**，那些会被覆盖的统计表变成"随时能从事件流重算出来的派生缓存"。怎么过渡到这一步，三个选项：

### 选项 D1：一次性重写
直接让事件流当真理源，统计表彻底改成开机/按需重算的缓存，旧的覆盖写路径删掉，历史数据一次性 backfill。
- **优点**：最干净、一步到位。
- **缺点**：一次性动 Swarm 核心写路径，**风险最高、回滚最难**，出问题难定位。和一/二期的增量纪律不符。

### 选项 D2（推荐）：双写过渡，分步切换
分三小步走，每步可停可回滚：
1. **并行追加**：在现有 Swarm 写路径**旁边**，新增一条只追加的协同事件流（不动现有统计表的任何写入）。
2. **影子验证**：让"一本账"投影从这条新事件流读 Swarm，与旧统计表做**对账**，确认两边讲的是同一个故事（过渡期读出可随时回退旧表）。
3. **切换降级**：对账稳定后，把"真理源"身份正式切到事件流，统计表降级为"可从事件流重建的读缓存"，最后才停止对旧表的直接信任。
- **优点**：每步独立可回滚、风险可控、能先拿对账证据再切换，与一/二期纪律一致。
- **缺点**：过渡期"半新半旧"并存一段；需要做一次对账才敢切。

### 选项 D3：只读不降级
Swarm 维持现状（统计表照旧可被覆盖），"一本账"投影只是顺带把它读出来拼进去，不做降级。
- **优点**：零风险、零改动。
- **缺点**：没真正兑现 ADR-022 的"旧表降级"，Swarm 轨迹仍可能被覆盖、复盘严谨性没提升——只是"把乱账读出来"，没"把账理顺"。

**推荐 D2。** 它是"增量、随时可停、先证据后切换"这套已被一/二期验证过的纪律在 Swarm 上的自然延伸；D1 太莽，D3 等于没做 ADR-022 想要的事。

> **本 ADR 只需你拍板 D2 这个**方向**；D2 的三小步具体实现放在 3b（届时给一份 3b 实施计划，仍是 TDD + 分层验证，你不看代码、只看对账证据）。** 3a 这条分支**不碰 Swarm 写路径**，只按 D3 的方式"只读拼进投影"——即先把 Swarm 以"读缓存"身份纳入一本账，降级动作留给 3b。

## 四、对第三期分段的指导

| 段 | 做什么 | 风险 | 受哪个决策点管 |
|---|---|---|---|
| **3a**（本分支 `feat/event-ledger-phase3`） | 读侧投影统一读出面 + 收编任务事件（已是 append-only）；Swarm 暂以只读身份拼入 | 低（纯只读、不动写路径/schema） | 决策点 1 = P2 |
| **3b**（后续分支） | Swarm 旧表降级 = D2 双写过渡三小步（并行追加→影子对账→切换降级） | 高（动 Swarm 写路径） | 决策点 2 = D2 |

> 3a 跑出"一本账复盘"招牌证据后，再评估是否立即启动 3b。沉没成本始终限于已完成段。

## 五、影响范围

- **用户可感知**：复盘/排障能"一本账查到底"（对话+任务+协同+成本+决策+执行 一条时间线）；日常使用无感。
- **数据层（3a）**：**不新增表、不改 schema、不改任何写路径**；仅新增一个只读投影服务 + 一个诊断读出口；两张账本仓储各补一个按会话查询的 SELECT 方法（仍 append-only）。
- **数据层（3b）**：新增一条 Swarm append-only 事件流；旧统计表逐步降级为派生缓存（双写过渡）。
- **不影响**：发版流程、模型路由、计费口径、现有 UI 交互、现有任何表的写入行为（3a 段）。
