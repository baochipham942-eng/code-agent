> **设计隐喻（绑定 Dream 的由来）**：人脑睡眠是一个统一的"离线维护窗口"，里面有巩固（consolidation，对应 Dream 提炼记忆）、修剪（pruning，对应 Dream prune 旧记忆）、和**一致性重整（reconsolidation，对应"对账"——把分散存储核对成自洽的一本账）**等子过程。对账与 Dream 同属"离线维护"，故**绑定 Dream 的节拍与触发**；但像大脑各子过程各司其职一样，对账是**独立的确定性步骤**（零 LLM、fail-safe），不塞进 Dream 的 LLM 推理、不与之连坐。

## 一、为什么补这个 ADR

ADR-022 §四把第四期定义为"上线后台对账（定期检查总账和明细对不对得上，发现偏差自动修）+ 迁移历史数据库"，但没给两个落地策略：

1. **后台对账这个"无人值守定时任务"，默认开到什么档位？** 它是整个项目里第一个会"自己定期跑、还可能自动改数据"的后台 job——必须服从林晨的「自动化产出验收三硬规则」：硬门代码化、无人值守不自动激活（要有显式开关/默认关）、运行证据硬门。不能做成"看起来在跑但没证据 / 没人盯着就偷偷改数据"。

2. **存量老库怎么升级？** 3b 之后，新跑的 Swarm run 都有了 append-only 真理源账本（ledger）；但**老 run 只有旧的 rollup 统计表、没有 ledger**。要不要在升级时把老 run 也"反向重建"出 ledger，还是安全跳过？这是迁移类改动，错了会毁老用户现有数据，必须明确"可回滚、幂等、向后兼容"。

这两点定了，第四期才能 TDD 推进、用户只看证据不看代码。

---

## 二、决策点 1：后台对账默认激活档位？（对应硬规则「无人值守不自动激活」）

> 通俗讲：对账分两件事——**只读的"查账+出报告"**（无害，只是读两边数据比对、写一份报告）和**会改数据的"发现对不上就重建缓存"**（这才是危险动作）。问题是这个定时 job 默认开到哪一档。

### 选项 A1（推荐，已拍板）：注册 + 开，但只报告
定时 job 默认注册并按计划跑，但**只做只读对账 + 出报告 + 落运行证据，绝不改数据**。"发现偏差自动重建缓存"是一个**单独的、默认 OFF 的写闸门**，要显式打开才会动数据。
- **优点**：等同项目已验证的 `memory-consolidation` dry-run 范式（调度照跑、只报告、写动作默认关）。能**无人值守持续产出对账证据**（满足"运行证据硬门"+ 可演示），同时**危险动作（写）默认关**（满足"无人值守不自动激活"——激活的是只读扫描，可激活的危险部分=写，默认关）。
- **缺点**：发现偏差后不会自动修，需要人看报告后显式开写闸门或手动重建。

### 选项 A2：注册但默认禁用（enabled:false）
job 注册成禁用态，什么都不跑，直到用户显式打开开关。
- **优点**：对"默认关"最严格的字面解读。
- **缺点**：激活前**没有任何对账证据 / 没法演示**；与 ADR-022"上线后台对账"的目标有距离。

### 选项 A3：纯手动触发
不注册定时 job，只保留按需触发（诊断出口 / 手动 run）。
- **优点**：最保守。
- **缺点**：没真正兑现 ADR-022"无人值守后台对账"。

**拍板 A1。** 理由一句话：**只读对账+报告是无害的、且是"运行证据硬门"和"可演示对账报告"的来源，应当默认跑；真正需要"默认关、显式开"的是会改数据的缓存重建——把它做成独立、默认 OFF 的代码级写闸门，恰好同时满足三条硬规则。**

## 三、决策点 2：存量老库（有 rollup、无 ledger 的老 run）迁移策略？

> 背景：3b 之后新 run 有真理源账本（ledger）；老 run 只有可能有损的 rollup 统计缓存。读路径 `getSwarmRunDetailPreferLedger` 本就对"无 ledger"的 run **自动回退读 rollup**，所以老 run 即使完全不迁移也能正常读、不会坏。

### 选项 B1（推荐，已拍板）：默认跳过 + 可选 backfill
升级时**不做任何破坏性动作**：老 run 继续靠读路径回退 rollup（已能正常工作）。另外提供一个**幂等 + 事务（可回滚）+ opt-in**的 backfill 工具（从 rollup 反向重建 ledger），需要把老 run 也统一成"一本账"时再手动/诊断触发。
- **优点**：升级零风险、绝不毁老数据；避免把"有损缓存"**自动**当成"真理源"enshrine；要统一时仍有路径。
- **缺点**：不主动触发的话，老 run 暂时仍以 rollup 身份读。

### 选项 B2：升级时自动 backfill 全部
开机自动给每个老 run 从 rollup 重建 ledger，立刻全量"一本账"。
- **缺点**：为全部历史用**有损缓存**重建"真理源"；且会让对账对这些 run 恒等通过（ledger 由 rollup 派生→循环自证），给假信心。

### 选项 B3：只跳过、永不 backfill
老 run 永远靠 rollup 回退，不提供重建路径。
- **缺点**：老数据永远拿不到真正的 ledger。

**拍板 B1。** 理由一句话：**读路径已对老 run 优雅回退，升级本身可以"什么都不破坏"；自动用有损缓存重建真理源（B2）有数据完整性与循环自证风险，永不重建（B3）又太死——默认跳过、把重建做成幂等可回滚的 opt-in，最稳。**

## 四、对第四期实施的指导

| 子项 | 做什么 | 风险 | 受哪个决策管 |
|---|---|---|---|
| 后台对账（只读扫描 + 报告 + 运行证据） | **绑定 Dream**：在 `dreamExecutor` 跑完记忆巩固后，fail-safe 调用独立对账服务，对账摘要附进 Dream 运行报告；频率/触发跟 Dream（每 7 天）走，report-only | 低（不动共享调度 dispatch；Dream 收尾薄调用、吞错不连坐） | 决策点 1 = A1 + Q3 绑定 Dream |
| 偏差自愈（发现 drift 自动重建 rollup 缓存） | 默认 OFF 的代码级写闸门（dryRun=false 才动）；仅当 ledger 为已闭合真理源时重建；**独立、不绑 Dream** | 高（动 Swarm 读缓存写路径 → 对抗审查 + TDD） | 决策点 1 = A1（写=默认关） |
| 老库迁移 | `backfillSwarmLedger` 幂等+事务+opt-in；默认跳过（不在开机/不随 Dream 自动跑）；**独立、不绑 Dream** | 高（迁移类 → TDD + 可回滚 + 向后兼容确定性验证） | 决策点 2 = B1 |

## 五、三硬规则如何在第四期落地（自动化产出验收）

1. **硬门代码化**：是否允许改数据 = 代码级 `dryRun` 闸门（默认 true=只报告），不是只靠配置；缓存重建只在 ledger 已闭合（含 run_closed）时执行，绝不用半套账本覆盖完整 rollup（沿用 3b HIGH-1 的修复不变量）。
2. **无人值守不自动激活**：会改数据的缓存重建默认 OFF；老库 backfill 默认跳过、不在开机/不随 Dream 自动跑（B1）。只读对账扫描随 Dream 默认开（它无害，且是证据来源）。
3. **运行证据硬门**：对账绑定 Dream → Dream（cron job，auto 跑会落 `cron_executions`）的 execution `result` 里带对账摘要；对账服务**自身**再归档一份可读报告（含扫描窗口/匹配/偏差/跳过）到固定位置 + `docs/plans/evidence/`。无运行记录的"光有定时器"不算数；对账即便 fail-safe 吞错，也必须留下"本次跑过、结果如何"的记录。

## 六、影响范围

- **用户可感知**：能定期拿到"总账 vs 明细缓存"的对账报告；老库升级无感、不丢数据。日常使用无感。
- **数据层**：不新增表（复用 3b 的 `swarm_run_ledger` + `cron_executions`）；不改任何现有写路径的默认行为（缓存重建写动作默认关）。
- **不影响**：发版流程、模型路由、计费口径、现有 UI 交互、现有任何表的默认写入行为。
