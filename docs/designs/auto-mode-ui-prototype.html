Agent Neo · ADR-019 · Design Prototype

自动模式改造 · 前端原型

核心思路一句话：把"黑盒路由"翻转成"白盒调度"。 业内 auto 模式（Cursor / GPT-5 Router）最大的失败不是算法，是用户不知道发生了什么。 Neo 反着做——每一次模型选择都看得见、看得懂、可撤销。

PRINCIPLE 01

白盒化

路由发生的那一刻就告诉用户：谁在干活、用什么模型、为什么。不需要打开任何面板。

PRINCIPLE 02

计费感知

"省钱"只对按量付费成立。包月用户的免费路由省的是 0 元——UI 要诚实地反映这一点。

PRINCIPLE 03

永不向上

自动模式永远不选比用户默认模型更贵的模型。这是写进代码的硬约束，UI 把它说出来建立信任。

A

聊天主链路 · 透明度 Trace（P1）

所有路由事件在聊天流里原位呈现，复用现有 MessageBubble / TaskPanel 结构，只新增 5 个轻量组件。 数据全部来自 P2 的结构化路由决策 {requested, resolved, reason, role, billingMode}——UI 不自己猜，只渲染决策结果。

自动 Kimi K2.5
1

路由 Chip（默认收起）——文案"自动 · 已用 GLM-4-Flash 回答 · 免费"，一行讲清三件事：这是自动决定的 / 用了什么 / 对钱包的影响。绿点=免费档、青点=主力、黄点=降级，末尾 ▸ 暗示可展开。默认只占一行，不打断阅读流。

2

决策详情卡（点击 chip 展开 / 再点收起）——标题用"为什么用这个模型"而不是"路由决策详情"（用户语言 > 系统语言）。关键是最后一行计费影响：按量付费显示真实节省，包月显示"套餐内 · 无额外成本"，绝不显示虚假的省钱数字。底部"本轮改用默认模型重答"——可撤销是信任的最后一环。

3

任务卡角色分层——系统 B（角色即路由）的可视化：每个子代理一行，模型标签用颜色区分档位。用户第一次直观看到"为什么探索便宜、写码用主力"。这块复用现有 TaskPanel 卡片，只是把模型标签从隐藏改为常驻。

4

降级横幅——可用性兜底触发时原位置插入聊天流（不是 toast，toast 会消失）。文案准确性：Neo 的消息历史是模型无关的，切 provider 不会丢上下文，只是回复风格可能有差异——横幅只说真话，不夸大风险。提供"等待恢复"选项，把决定权还给用户。

5

StatusBar 保持现状——只保留现有的"自动"徽标 + 默认模型名，不加本轮路由结果和省钱金额（信息已在 chip 和决策卡中，StatusBar 重复展示是噪音）。

B

设置页 · 计费语义四分类（P2）

复用现有"模型设置"master-detail 布局，在 Provider 详情里新增计费方式一栏。 这是整个取舍方案的地基：simple 路由门控、成本统计、决策卡的计费影响全部从这个用户配置读取，不再依赖手工维护的价格表猜测。

6

计费方式四分类——下拉只有四个选项，每个都用人话解释。重点：这是用户配置而不是系统猜测。市场价（来自价格库）和"你的 Key 怎么计费"（用户配置）从此分离。默认值贴近现实：新 Key 默认按量付费（API Key 的主流形态，省钱路由默认生效），中转站默认"未知"（保守）。

7

门控开关 + 智能默认——三个开关对应 adaptive 拦截器的三个职责（省钱 / 能力 / 可用性）。关键交互：简单任务开关的可用性由计费方式决定——包月用户看到的是"已自动关闭 + 解释原因"，而不是一个骗自己的开关。这就是"计费感知"原则落在 UI 上的样子。

8

硬约束声明——用虚线框 + 锁图标 + "不可关闭"标签做视觉隔离，刻意和普通设置项区分开。它不是设置，是承诺。面试讲这个细节：产品如何用 UI 语言传达"这是底线不是选项"。

9

价格档位 + 数据源——回答"Opus 一定比 DeepSeek 贵，系统怎么知道"：路由约束只需要比大小，所以用 5 档价格带（免费 / $ / $$ / $$$ / $$$$），不依赖精确数字。数据源是 LiteLLM 社区价格库（业内通用），打包内置快照 + 每周自动更新，分发给任何用户都不需要手工维护价格表。Provider logo 复用项目已有的 providerLogoCatalog.ts，不是字母占位。

C

路由决策流程 · 单一入口（P2 核心）

最复杂的部分：两套系统（角色分层 / adaptive 拦截器）如何在一个决策入口里协作， 消除现在"两条引擎、两种行为"的问题。所有路径殊途同归到同一个结构化决策对象，UI / 日志 / 成本统计只消费这个对象。

9

为什么必须是"单一入口"——现在的 bug（adaptive 泄漏、两引擎不一致）根源是路由决策散落在 4 个地方各自为政。收口后：决策逻辑只有一份，AI SDK 引擎和 legacy 引擎都调用它，行为不可能再分叉。

10

系统 B 在左、系统 A 在右——subagent 任务直接走角色映射，根本不经过 adaptive 拦截器（彻底解决泄漏问题，而不是修补泄漏）。只有主聊天 + 用户开了自动，才进入右侧三个检查。

11

硬约束门是所有路径的必经之路——无论从哪个分支出来，切换动作都要过"不比默认贵"这道门。把它画成虚线红框 + 锁，对应设置页里"不可关闭"的声明——代码、流程、UI 三处说同一句话。
