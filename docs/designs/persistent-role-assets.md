# 持久化角色资产（Persistent Role Assets）设计方案

> 状态：设计定稿，待实现（MVP 范围见 §8）
> 日期：2026-06-03

> 决策记录：见 §12（2026-06-03 与林晨确认）

---

## 1. 背景与定位

### 1.1 要解决的问题

Neo 的 agent 今天是**失忆临时工**：每次 spawn 都是入职第一天，不知道用户的业务、不知道上次踩的坑、不积累任何东西。全局 Light Memory 解决了"用户级"记忆，但没有"角色级"记忆——"数据分析师"这个角色第 100 次被调用和第 1 次一样笨。

### 1.2 核心论点

**持久的是资产（角色定义 + 记忆 + 履历），瞬时的是实例（运行中的 agent）。角色和记忆是户口，实例是上班。**

这个设计同时吸收了两个对立观点的合理部分：

- Slock/Cumora 的"持久队友"：有价值的不是 agent 常驻运行，而是它的记忆持久存在
- 黄佳的终局判断（《Claude Code 工程化实战》14b/08）："Harness 根据情况自动创建并清理 Agent"、"skills/subagents 作为能力插件"——实例是瞬时的

### 1.3 为什么这是 Neo 的护城河

1. **经验累积 = 用户切换成本**：模型是租来的，角色资产是用户自己的。"我的数据分析师对我业务的理解"搬不到竞品里
2. **本地差异化**：资产是用户机器上的文件，所有权和隐私归用户——云产品（Coze/MuleRun/WorkBuddy）的资产锁在平台账号里
3. **成本结构**：不为闲置付费（vs Slock 常驻 daemon 模式）

## 2. 核心概念与原则

| 概念 | 定义 |
|------|------|
| **角色（Role）** | 持久资产 = 角色定义 + 角色记忆 + 工作履历。文件形态，永远存在 |
| **实例（Instance）** | harness 按需拉起的运行进程。无状态，用完即弃 |
| **写回（Write-back）** | 实例销毁前把值得记的东西落盘到资产。这是让循环成立的纪律 |

**三原则**：资产是文件（git-friendly、可审计、可分享）/ 实例无状态（可随时杀掉重开）/ 写回是纪律（无写回 = 失忆实例）。

## 3. 三层记忆架构（决策：一步到位）

```
全局记忆（用户级）          ~/.code-agent/memory/              ← 现有 Light Memory，不动
角色记忆（角色级）          ~/.code-agent/roles/<role>/         ← 本方案新增
项目记忆（项目级）          ~/.code-agent/projects/<key>/memory/ ← 本方案新增（过渡形态）
```

### 3.1 三层的内容边界

| 层 | 装什么 | 例子 | 谁读 |
|----|--------|------|------|
| **全局** | 用户身份、偏好、跨一切的事实 | "用户是 PM，讨厌套话，回复用中文" | 所有实例 |
| **角色** | 该角色的专业积累，跨项目通用 | 数据分析师："用户的业务口径：GMV 不含退款" | 该角色的实例 |
| **项目** | 该项目特定的上下文 | "这个项目的产物是周报，模板在 docs/templates/" | 在该项目工作的所有实例 |

### 3.2 写回路由规则（write gate 的一部分）

实例结束时模型对每条候选记忆做三选一判断：

- 和具体角色无关、和具体项目无关 → **全局**
- 是这个角色的专业知识，换个项目还有用 → **角色**
- 只在这个项目里有意义 → **项目**

判断不确定时降级到更窄的层（项目 > 角色 > 全局），避免污染宽层。

### 3.3 读取注入规则

实例化时注入 = 全局 INDEX（现有）+ 角色 MEMORY.md + 当前项目的记忆索引。三层都走"索引常驻 + 正文按需"的渐进披露模式（复用 Light Memory 现有机制）。

### 3.4 项目层的过渡方案

P0-2 项目空间还没做，项目层先用 **workspace 路径**作为 key（`~/.code-agent/projects/<workspace-path-hash>/`，目录里存一个 `meta.json` 记录原始路径）。P0-2 落地后 Project 实体接管这个 key，记忆文件不动、只换索引——这就是"预留不迁移"。

## 4. 角色资产模型

### 4.1 文件结构

```
~/.code-agent/agents/数据分析师.md         ← 角色定义：现有 custom agent 格式，完全不动
~/.code-agent/roles/数据分析师/             ← 角色资产目录：按名字绑定（约定优于配置）
├── MEMORY.md                              ← 角色记忆索引（~200 行上限，注入用）
├── memories/
│   ├── user-business-glossary.md          ← 记忆文件，frontmatter 含 type/created/expires/project
│   └── ...
└── history.md                             ← 工作履历
```

### 4.2 绑定机制（决策：agent 定义零改动）

- `agents/*.md` 保持 Claude Code 兼容格式，**一个字都不改**
- 运行时按名字检测 `roles/<同名>/` 目录：存在 → 这是"持久角色"，注入资产；不存在 → 普通瞬时 agent，行为和今天一样
- 渐进式：任何现有 agent 加一个目录就升级成角色，删掉目录就降回去

### 4.3 工作履历 = 产物清单（产物中心总纲的体现）

`history.md` 的条目不是"做过什么任务"，是"参与过什么产物"：

```markdown
- 2026-06-03 | [Q2 用户增长周报](artifact://deck/xxx) | 产出 12 页 Deck | 用户反馈：图表口径改了一次
- 2026-06-01 | [竞品数据看板](artifact://dashboard/yyy) | 迭代 3 轮定稿 | -
```

每条指向一个 artifact ID。项目空间（P0-2）落地后，履历自动按项目分组。

## 5. 生命周期循环

```
┌─ 1. 实例化 ──────────────────────────────────────────┐
│  SubagentExecutor.spawn(agentName)                    │
│  → agentRegistry 查角色定义（现有）                     │
│  → 检测 roles/<name>/ 存在？                           │
│  → 是：注入 角色 MEMORY.md + 项目记忆索引（新增一层）     │
├─ 2. 工作 ────────────────────────────────────────────┤
│  正常 agent loop（不动）                               │
│  工作中可调 MemoryWrite 实时写（现有工具，加 scope 参数） │
├─ 3. 写回 ────────────────────────────────────────────┤
│  实例结束（正常完成 / Stop hook 时机）                  │
│  → 模型自判断"这次有什么值得记的" + 路由到层（§3.2）      │
│  → write gate：去重检查（grep 现有记忆）+ 质量门槛       │
│  → 更新 MEMORY.md 索引 + history.md 履历               │
├─ 4. 销毁 ────────────────────────────────────────────┤
│  实例清理（现有）                                       │
│  异常退出兜底：taskPatchService（PR #203）保住工作现场   │
└──────────────────────────────────────────────────────┘
```

### 5.1 写回质量保障（write gate）

复用全局 Light Memory 已验证的纪律，外加：

1. **去重**：写入前 grep 该层现有记忆，相关的更新而不是新建
2. **质量门槛**：拒绝"完成了 X 任务"这类流水账（履历管这个），只收"下次还有用"的知识
3. **配额**：单次实例写回 ≤ 3 条，防止记忆灌水
4. **用户可见可改**：UI 上每条记忆可删可编辑（§7）

## 6. 完整角色系统（产品全貌）

> MVP 只做 §6.1 的 2 个预设角色，但产品全貌按此设计，后续阶段不返工。

### 6.1 预设角色（builtin roles）

随产品分发，开箱即用。第一批两个（MVP）：

| 角色 | 定位 | 差异化能力 |
|------|------|-----------|
| **研究员** | 调研、信息收集、报告产出 | 绑定 web/search 工具组 + 调研类 skills |
| **数据分析师** | 数据处理、看板、周报 | 绑定 Excel/chart 工具组 + 数据类 skills |

预设角色的"角色定义"随版本更新，但用户机器上的"角色记忆"永远是用户自己积累的——**定义是出厂设置，记忆是使用痕迹**。

### 6.2 用户自建角色

任何 custom agent + 一键"升级为持久角色"（= 创建 roles/ 目录）。入口在角色面板。

### 6.3 角色分享 / 导出（Phase 3）

打包 = 角色定义 + 记忆**模板**（可选勾选哪些记忆进包，默认不含个人数据）。对应 Codex role plugin 的"装备包"概念，但多了"可以带经验出厂"。

### 6.4 角色市场（远期，不排期）

社区分享角色包。与 Capability Center 现有 registry 机制合流。

## 7. UI 设计（角色面板）

复用 MemoryTab 文件浏览器模式（现有），新增"角色"入口：

```
角色面板
├── 角色列表（预设 + 用户自建，卡片：头像/名字/记忆条数/最近工作）
└── 角色详情
    ├── 定义（只读展示 agents/*.md，编辑跳转文件）
    ├── 记忆（分层标签：角色记忆 / 涉及的项目记忆，可删可编辑）
    └── 履历（产物清单，点击跳转 artifact）
```

聊天输入区：@角色名 唤起（现有 agent 选择机制，标持久角色徽标）。

## 8. MVP 范围（第一期实现）

**做**：
1. 三层记忆的目录结构 + 读写路由（§3）
2. 角色绑定机制（§4.2）+ 履历写入（§4.3）
3. 生命周期循环完整跑通（§5），含 write gate
4. 2 个预设角色（研究员 + 数据分析师）
5. 角色面板 UI 最小版（列表 + 详情 + 记忆删改）
6. MemoryWrite/MemoryRead 工具加 scope 参数（global/role/project）

**不做（但接口预留）**：
- 主动性 cadence 触发器（实例化入口设计成可被 hook 调用的形式，见 §9）
- 用户自建角色的 UI 引导（手动建目录即可用，UI 下期）
- 角色分享/导出
- 项目空间实体（项目记忆用 workspace key 过渡）

**验收标准（E2E，非单测）**：
1. 用研究员角色做一次调研 → 角色记忆出现合理条目 → 关掉应用重开 → 再用研究员 → 它引用上次的记忆
2. 同一个用户换数据分析师角色 → 看不到研究员的角色记忆，但看得到全局记忆
3. 同一角色在两个不同 workspace 工作 → 项目记忆互相隔离
4. 角色面板能看到记忆和履历，删除一条记忆后实例不再引用它

## 9. 主动性接口预留（本期不实现）

实例化入口统一为：

```typescript
instantiateRole(roleName: string, trigger: 'user' | 'cadence' | 'event', context: InstantiationContext)
```

本期只有 `trigger: 'user'` 路径。下期主动性 = Hook 系统加 cadence 触发器 → 调同一个入口传 `'cadence'` → 角色带着记忆醒来 → 检查产物状态 → 推进/汇报/沉默 → 写回销毁。

## 11. 风险与开放问题

| 风险 | 应对 |
|------|------|
| 写回质量差 → 资产污染 → 角色越用越笨 | write gate 三道闸（去重/门槛/配额）+ 用户可删改 + MVP 期间人工抽查写回内容 |
| 三层路由判断增加每次实例结束的成本 | 写回用 quick model（成本可忽略）；判断 prompt 给明确规则减少思考量 |
| 角色记忆和全局记忆内容重叠 | 写回路由规则"不确定时降级到窄层"+ 定期整理 cron（复用 WS4-B) |
| 项目 key 过渡方案（workspace hash）在 P0-2 时迁移 | meta.json 记录原始路径，迁移只换索引不动文件 |
| 多实例并发写回同一角色记忆冲突 | MVP 阶段串行写回（SerialWriteGate 已有类似机制）；后续再考虑合并策略 |

## 12. 决策记录（2026-06-03，与林晨确认）

| 决策点 | 结论 | 备注 |
|--------|------|------|
| 记忆层级 | **三层一步到位**（全局/角色/项目） | 项目层用 workspace key 过渡，P0-2 落地后接管 |
| 角色实体 | agent 定义零改动 + roles/ 目录按名字绑定 | 保持 Claude Code 格式兼容 |
| 写回机制 | 模型自判断 + write gate + 用户可删改 | 复用已在 Claude Code memory 体系验证两个月的模式 |
| MVP 范围 | **2 个预设角色 e2e，方案按完整系统设计** | 验收标准见 §8 |
| 主动性 | 下一期，本期只预留 instantiateRole 接口 | 先验证资产质量再做主动唤起 |
| 与 P0-2 顺序 | 角色资产先行，项目空间下期 | 角色后续"入驻"项目 |
