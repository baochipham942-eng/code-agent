# v0.16.79 — webServer auth broadcast 修复（管理员菜单真正回填）

> 2026-05-19

v0.16.78 修了 webModeWindow 注册到 main/app/window.ts 的 mainWindow（让 SSE bridge 通），
但实测管理员菜单还是不出现。继续 debug 发现**更深根因**。

## 根因（v0.16.78 没修到）

`mainWindow.webContents.send(EVENT_CHANNELS.AUTH, ...)` 的代码在
`src/main/app/initBackgroundServices.ts:201` 的 `addAuthChangeCallback` 里。

但这个 bootstrap 是 **Tauri 桌面 main 进程** 的入口
（src/main/app/bootstrap.ts → initializeBackgroundInfra）。

**web 模式**（实际所有 release 都是 Tauri shell + webServer Node 子进程的形态）
启动走的是 `src/web/webServer.ts`，里面**只调了 `authService.initialize()`，
完全没注册 broadcast callback**。

结果：fetchUserProfile 第二次成功（带 isAdmin: true）后，notifyAuthChange 跑了，
但 onAuthChangeCallbacks 是空的，没人 broadcast，renderer 永远收不到。

v0.16.78 改的 setMainWindow 让 BrowserWindow.webContents.send 能走 SSE，
但 **callback 这一头根本没注册**，等于水管两头一头通了一头没。

## Fix

`src/web/webServer.ts:417` 初始化 AuthService 之前显式 `addAuthChangeCallback`：

```ts
authService.addAuthChangeCallback((user) => {
  broadcastSSE('auth:event', {
    type: user ? 'signed_in' : 'signed_out',
    user,
  });
});
await authService.initialize();
```

这才是真正把 main 端 auth 变更推到 renderer authStore 的链路。

## 链路总结

```
fetchUserProfile (Supabase) 成功
  → notifyAuthChange(user with isAdmin)
  → onAuthChangeCallbacks.forEach
  → 我们注册的 callback
  → broadcastSSE('auth:event', {type, user})
  → 所有 /api/events 长连接
  → renderer ipcService.on(IPC_CHANNELS.AUTH_EVENT)
  → authStore.setUser
  → Sidebar 看到 user.isAdmin = true
  → 管理员菜单显示
```
